# Password Strength Predictor API

Predicts whether a password is **Weak**, **Medium**, or **Strong**, served through
a REST API built with FastAPI. Trained on ~670K labeled passwords using
character-level TF-IDF features.

Built to practice taking an ML model beyond a notebook — wrapping it as a real
service, the way it would be used in a signup form or password manager.

## Results

| Model | Accuracy | Weighted F1 |
|---|---|---|
| Logistic Regression | 85.9% | 0.84 |
| **Random Forest (used)** | **89.5%** | **0.88** |

### A limitation worth calling out

Aggregate accuracy looks strong, but the **per-class breakdown** tells a more
useful story:

| Class | Precision | Recall | F1 |
|---|---|---|---|
| Weak | 0.96 | **0.35** | 0.51 |
| Medium | 0.88 | 1.00 | 0.93 |
| Strong | 0.98 | 0.89 | 0.93 |

The model only **catches 35% of genuinely weak passwords** — it's biased toward
predicting "Medium" for borderline cases. Concretely: `qwerty` and `123456`,
two of the most common weak passwords in existence, both get classified as
"Medium" by this model. The 89.5% headline accuracy is real, but it's propped
up by the large Medium class (496K of 670K rows) being easy to get right — the
class that actually matters for a security tool is the one the model is
weakest on.

This is why `/model-info` documents it explicitly, and why the README's "what
I'd add next" includes combining this with rule-based checks rather than
relying on the model alone for anything security-critical.

## What it does

- Trains a classifier on ~670K passwords labeled by strength
- Uses **character-level** TF-IDF (not word-level — passwords aren't made of
  words) so the model learns patterns in character composition and sequence
- Compares Logistic Regression vs Random Forest, keeps the better one
- Serves predictions through a FastAPI endpoint with full class probabilities

## Tech used

- Python, scikit-learn, pandas
- FastAPI + Pydantic for the API
- joblib for model persistence

## A bug worth mentioning

The original notebook defined `char_tokenizer` inline. That works fine inside
the notebook, but `TfidfVectorizer` pickles a *reference* to the tokenizer
function, not the function itself — so loading `vectorizer.pkl` from any
other process (like the API) failed with:

```
AttributeError: Can't get attribute 'char_tokenizer' on <module '__main__'>
```

Fixed by moving `char_tokenizer` into `features.py`, a real importable module
that both the training script and the API import from. Small fix, but it's
the difference between "works in my notebook" and "works as a deployed
service" — exactly the kind of thing that's invisible until you actually try
to serve the model.

## Running locally

```bash
pip install -r requirements.txt

# Train the model (downloads nothing -- uses data/Password_Strength.csv)
python notebooks/01_train_model.py

# Start the API
cd api
uvicorn main:app --reload
```

Open `http://127.0.0.1:8000/docs` for interactive Swagger UI.

### Example request

```bash
curl -X POST "http://127.0.0.1:8000/predict" \
  -H "Content-Type: application/json" \
  -d '{"password": "Xk9#mPz2Qw!"}'
```

### Example response

```json
{
  "password_length": 11,
  "strength_label": "Strong",
  "strength_score": 2,
  "confidence": 0.718,
  "class_probabilities": {
    "Weak": 0.026,
    "Medium": 0.256,
    "Strong": 0.718
  }
}
```

## Endpoints

| Method | Path | Description |
|---|---|---|
| POST | `/predict` | Predict password strength with full class probabilities |
| GET | `/model-info` | Model metadata, scores, and known limitations |
| GET | `/health` | Health check |

## Note on model file size

`model.pkl` is ~48MB (100-tree Random Forest). A smaller config (75 trees,
shallower depth) was tested to cut this to ~13MB, but it dropped Weak-class
recall from 0.35 to 0.14 — too large a hit for a password-strength model,
where catching weak passwords is the entire point. Kept the larger, more
reliable model and use **Git LFS** to track it instead:

```bash
git lfs install
git lfs track "models/*.pkl"
git add .gitattributes models/*.pkl
```

## Dataset

Public password-strength dataset from Kaggle — password + strength label,
~670K rows.

## What I'd add next

- Combine the ML prediction with rule-based checks (length, digit/symbol
  variety) specifically to catch the weak passwords the model currently
  misses — a hybrid approach would directly address the recall gap above
- Deploy with a live URL instead of localhost-only
- Add basic rate limiting since this would sit behind a signup form in
  practice
